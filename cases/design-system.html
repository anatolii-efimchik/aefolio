Context

How it started

A B2B platform had been growing for 2.5 years without a shared design system. Three product teams built independently — each in their own Figma file, each with their own interpretation of what a button, a card, or a form should look like.

214 unique color values. 54 button variants. 3 teams. 0 shared components.

Three products of one company looked like they were made by three different companies.

Every sprint added new inconsistency debt: QA found style bugs that were actually design decisions, developers rebuilt what another team had built a month ago, and designers spent hours searching for 'the right button' instead of solving product problems.

The real problem wasn't that the interface looked inconsistent. The problem was that teams couldn't collaborate — each spoke a different visual language.

Process

What I did and in what order

I led the design system work from scratch — as the single point of responsibility for the cross-team design infrastructure.

Stage What I did
Audit Inventoried 12 Figma files, found 400+ unique components and 214 color values
Token architecture Designed a three-level token hierarchy: global → semantic → component
Core library Built 40 base components with variants, states and auto-layout
Documentation Wrote usage guidelines: do/don't, context of use, connections between components
Adoption Conducted workshops, migration guide, weekly design syncs across teams

Audit

What I found in the files

Before designing anything, I spent two weeks auditing all existing design files. Without understanding the scale of chaos, any solution risks solving the wrong problem.

Problem area What was found Effect
Color 214 unique hex values, 6 shades of 'brand blue' Dark theme impossible, theming requires full rebuild
Typography 28 text styles, no semantic naming Developers hardcoded font sizes, no shared type scale
Buttons 54 variants across 3 teams 3 implementations in code, endless PR conflicts
Spacing Free-form — any value from 4 to 48px Pixel-perfect handoff impossible without manual annotation
Icons 3 icon sets mixed together Different visual weight of icons within one product
// тут вставить изображение · DS-02-audit.png — таблица аудита из Figma

The audit showed: the problem isn't too many designers. The problem is there was no contract between teams. Everyone worked as best they could — but by their own rules.

Architecture

How the system is organized

The first and most important decision: start with tokens, not components. A token is a named design decision — not '24px', but 'spacing-lg'. Not '#2563EB', but 'color-action-primary'.

Level Example What it does
Global blue-500: #2563EB Raw values — not used directly in design
Semantic color-action-primary: blue-500 Intent — what this value means in context
Component button-primary-bg: color-action-primary Component-specific binding — enables isolated overrides
// тут вставить изображение · DS-03-tokens.png — схема иерархии токенов

Why tokens first?

If you start with components, tokens become an afterthought — and theming, white-labeling, or dark mode will require rebuilding everything from scratch. Tokens are the foundation that makes the system extensible.

Not taken: starting directly with components — faster at first, but you pay double when a second theme or white-label version is needed.

Components

What went into the library

I chose a 'lean core' approach deliberately: not 200 components, but 40 well-thought-out ones covering 80% of all product screens. Each component has states, variants, auto-layout and documentation.

Group Components States
Actions Button, Icon button, Link Default, Hover, Active, Disabled, Loading
Inputs Input, Textarea, Select, Checkbox, Radio, Toggle, Date picker Empty, Focus, Filled, Error, Disabled
Feedback Toast, Alert, Badge, Tooltip, Skeleton Info, Success, Warning, Error
Navigation Tabs, Breadcrumb, Sidebar, Pagination Active, Inactive, Disabled
Data display Card, Table, Tag, Avatar, Empty state Default, Hover, Selected, Loading

Documentation

How we agreed on usage

The mistake most design systems make: comprehensive documentation that nobody reads. I chose a different principle — 'lean docs': only what's non-obvious, only where a decision needs justification.

  • Each component has one core usage rule (Primary — for the main action on a screen, one per screen)
  • Do / Don't examples directly in Figma — visible without switching context
  • Connections between components — when Toast vs Alert, when Tooltip vs Popover
// тут вставить изображение · DS-05-docs.png — страница документации с Do/Don't

A design system without documentation is just a library of beautiful frames. Documentation is what turns a system into a shared language — not a personal collection.

Adoption

How the system took root

The hardest part of any design system isn't building it — it's adoption. I had three teams with their own momentum and 2.5 years of accumulated patterns.

What I did Goal
Onboarding workshop for each team (2h) Show the system live, answer questions, remove fears
Migration guide with priority levels Teams migrated gradually without pausing product work
Weekly 30-min design syncs Surface blockers early, collect change requests
Office hours for developers Close the gap between Figma tokens and CSS implementation
// тут вставить изображение · DS-06-adoption.png — timeline внедрения по командам

Key insight from adoption: designers switched to the system in 2 weeks, developers — in 6. The bottleneck wasn't design — it was synchronizing tokens between Figma and the code repository.

Principles

Principles that guided me

Design system work taught me: the hardest thing isn't making something beautiful — it's making it useful to people who didn't participate in its creation.

1. Tokens before components

A design system without a token foundation is a beautiful thing you'll have to throw away when the first theming request arrives.

2. Lean core, not comprehensive coverage

40 well-documented components beat 200 poorly documented ones. A system people don't use is just a collection of frames.

3. Adoption is a design problem

Building the library is 40% of the work. 60% is making other teams actually want to use it. Workshops, syncs, office hours — this is part of design system work, not overhead.

4. A design system is never finished

It's a living product that grows with the teams using it. 'Version 1.0 is done' isn't the finish line — it's the start of the maintenance cycle.
