Case 02

Design system

UI language for a B2B platform — from 200 color values and 50 button variants to a shared library used by 3 teams and 12 engineers.

Format

TL;DR

For those who scroll fast

AdCombo is a CPA network with a large personal cabinet (offers, campaigns, statistics, payments, orders). Over the years the product accumulated dozens of screens built by different people at different times. There was no design system — the interface had become chaos: 9 shades of "brand" orange, 4 different button styles, forms rebuilt from scratch on every page.

I conducted an audit, assembled a design system (foundations → tokens → components), connected Figma to code, and migrated the key screens.

Metric Result
New screen release time −40% (~5 to ~3 days, design + dev)
Visual inconsistencies in new releases −85% (per QA logs)
Interface coverage by system components 0% → 72%
Sources of truth for design and code 1 (instead of scattered mockups)
// DS-01-metrics.png — 4 крупных числа-плашки (−40%, −85%, 72%, 1 source of truth)

Context

What was there

AdCombo is a partner network. The web cabinet is a dense, data-heavy product: offer tables, campaign builders, statistics with dozens of filters, payments, orders, settings. Plus a marketing side: registration landing pages and a blog.

No design system existed. I confirmed this with a technical audit — exported actual CSS from all key pages. The numbers speak for themselves:

Area What was found
Color The "primary" accent was different on almost every page: #e74011 on dashboard/news/payouts, #e63e21 on create-campaign/offer/settings, #ffbb0f on offers, #34a835 on statistics, #ff0000 on the landing — while the logo brand color is #ef4e23. Impossible to tell which color meant "action" and which meant "danger".
Grid Base spacing floated between 2/5/10/15px depending on the page — no unified spacing system.
Radii Button corner radii: 0, 3, 4, 15, and 50px — mixed even within a single screen.
Components A button existed in a dozen incompatible implementations (filled, filled-2…5, ghost, outline) with different padding, font sizes, and colors. Forms rebuilt from scratch on every page.
Typography Headings rotated between Scada, Inter, and Oswald; in the cabinet one font carried both headings and body text with no hierarchy.
Technical debt The :root contained an unconfigured default PrimeNG theme (--primary-color: #007ad9 — a blue absent from the brand) on top of custom --ADC--* variables. The token infrastructure existed but was never aligned to the brand.
Process Developers didn't know which mockup was "correct" and often coded from memory. Design and code diverged.
// DS-02-chaos.png — коллаж «свидетельства хаоса»: форма регистрации с лендинга и форма смены пароля из настроек рядом

Why it was a business problem, not taste

  • Every new screen = reinventing basic elements → slow releases.
  • Inconsistency reduced partner trust in the platform — a finance product where trust = money.
  • QA constantly caught visual bugs that weren't bugs — nobody knew the standard.
  • Onboarding new designers and developers took weeks.

Goals

What we set out to achieve

Business goal: speed up interface delivery and reduce the cost of UI maintenance. Product task: create a unified design system — a single source of truth for design and code — and migrate the key cabinet screens to it.

KPI Target
Time-to-market for a new screen Reduce
Visual inconsistencies in releases Reduce
Coverage of key screens by system components ≥ 70%
Design and code token alignment 1 shared token set

Scope constraints

Can't stop product development — migration is gradual. No rebranding — working with the existing identity (orange/yellow logo). Deliberately out of scope for phase 1: light/dark themes and animation/motion — to avoid inflating the system before the foundation is in place.

Research

UI inventory

I took screenshots of all cabinet and marketing screens and collected them on a single FigJam board. Then went through with a marker: highlighted every button, field, badge, table, color.

// DS-03-inventory.png — FigJam-доска с UI-инвентарём: сетка скриншотов с цветными пометками

What the inventory found — confirmed by CSS export (228 custom CSS variables in :root):

Found Needed
7+ shades of the primary accent across pages 1 brand palette
Dozen incompatible button implementations (filled, filled-2…5, ghost, outline) 4 roles
3 font families (Scada / Inter / Oswald) 2
Base spacing 2/5/10/15px 1 scale
Radii 0/3/4/15/50px 1 scale
PrimeNG default theme in :root, not aligned to the brand Brand tokens

Team interviews

Short interviews with 3 designers and 3 developers. Key insights:

  • Designers spent up to 30% of their time recreating basic elements.
  • Developers didn't trust mockups and asked about every spacing value.
  • Everyone wanted "a single set" but never had the resources to build it.
// DS-04-quotes.png — 3 карточки с цитатами из интервью

Benchmarking

Studied how public design systems are structured (Material, Atlassian, Polaris, Carbon) — how they build token hierarchies, how they document components, how they handle governance. From this I assembled the target architecture.

Strategy

Architecture

I chose a layered architecture: Foundations → Tokens → Variables → Atoms → Molecules → Organisms → Templates, with a three-level token system (primitive → semantic → component) and a Figma ↔ code connection via Code Connect.

// DS-05-architecture.png — пирамида слоёв со стрелками наследования

Foundations and tokens

First step: fix the foundations and clean up color. Key brand decision:

  • orange = brand and primary action
  • red = danger/error + primary marketing CTA
  • yellow = accent/label (TOP)

Reduced 9 shades to a clean palette with three token levels, verified contrast against WCAG AA.

Level Example What it does
Primitive color.orange.500: #EF4E23 Raw values — not used directly in design
Semantic color.action.primary: orange.500 Intent — what this value means in context
Component button-primary-bg: color.action.primary Component-specific binding — enables isolated overrides

Brand palette derived from the logo: orange-red #EF4E23 (primary), golden-yellow #FCB813 (accent/TOP labels), clean red #E01818 (errors). Typography normalized to two families: Oswald (headings/UI) + Scada (body text).

// DS-06-tokens.png — страница Foundations в Figma: палитра со свотчами и токенами, типошкала, spacing-scale

Components

Built the library: atoms (Button, Input, Select, Checkbox, Toggle, Badge, Tag…), molecules (Form field, Search, Pagination, Date field, Pill-tab…), organisms (Sidebar, Topbar, Data table, Filter panel, Offer card, forms, Date range picker).

Every component has all states: default / hover / active / focus / disabled / loading / error / empty.

// DS-07-components.png — сетка вариантов Button со всеми состояниями и размерами
// DS-08-table.png — организм Data table с вариантами: обычная строка, hover, выделение, пустое состояние

Figma ↔ code connection

Set up a code library on Tailwind + shadcn/ui, exported Variables to tokens.json via Tokens Studio, and connected Figma components to code via Code Connect — the actual snippet is now visible from the mockup.

In parallel made components AI-ready: wrote machine-readable metadata for each (variants, states, tokens, do/don't rules, anti-patterns) so AI assistants assemble interfaces from existing components rather than generating new ones. Tokens exported to a single tokens.json, metadata in a per-component schema.

// DS-09-code-connect.png — панель Code Connect: Figma-компонент Button рядом с его кодовым представлением

Screen migration

Migrated key cabinet screens (Offers, Statistics, forms, cabinet layout) to system components — a real-world stress test.

// DS-10-before-after.png — до/после: страница Offers — слева старая версия с разнобоем, справа пересобранная на компонентах системы

Documentation

Governance

Used a single template to document every component: anatomy, variants, tokens, do/don't, accessibility. Set up a changelog, semver versioning, and a contribution process (how to propose a new component).

A design system without documentation is just a library of beautiful frames. Documentation is what turns a system into a shared language — not a personal collection.

Qualitative effects

  • Designers stopped recreating basic elements and focused on product tasks.
  • Developers code faster without back-and-forth — mockup = code.
  • QA has a standard: "correct" now has a definition.
  • Onboarding new team members is faster.

Figures are based on team estimates (designer hours, developer story points) and QA logs — not formal A/B testing. Results are directional, not exact.

Learnings

What I learned / what I'd do differently

Tokens first, then components

Cleaning up color and typography removed up to 80% of visual chaos before any component was assembled.

Governance over aesthetics

A system only lives if there's a contribution process and an owner. Without it, it drifts apart again.

Migration is a product

I prioritized screens by frequency (tables, forms) rather than "in order" — fast visible impact over completeness.

What I'd do differently

Bring developers into the token work earlier — parallel rather than sequential design and code assembly would have saved weeks.
